# Context 1.2 — Учасник 2 (Domain/Integration & QA)

## 1) Загальний контекст практичної
- Практична робота №1: e-commerce Web API за класичними принципами проєктування.
- Тема команди: ринок верхнього одягу та кросівок.
- Реалізація на Python (FastAPI), з фокусом на ті ж принципи, що у вимогах (.NET-орієнтованих): SOLID, layered architecture, repository/service/DTO.
- Віртуальне оточення та менеджер залежностей для всієї команди: uv (обов'язково).

## 2) Роль Учасника 2 у команді
Учасник 2 відповідає за:
- моделювання і реалізацію сутностей, окрім core Product CRUD;
- API для Catalog/Customer/Order (мінімум GET + POST, де потрібно);
- інтеграцію модулів з роботою Учасника 1;
- тестування інтеграційних сценаріїв на рівні бізнес-потоків;
- валідацію цілісності даних і базові нефункціональні перевірки.

Послідовність виконання:
- Учасник 2 починає роботу тільки після повного завершення блоку Учасника 1.
- Учасник 2 бере другу половину робіт: Catalog/Customer/Order, інтеграція та фінальна стабілізація.

## 3) Ключові сутності в зоні Учасника 2
- Catalog/Category
- Customer
- Order
- (опційно) OrderItem

Мінімальні зв'язки:
- Catalog має багато Product
- Customer має багато Order
- Order має багато OrderItem
- OrderItem посилається на Product

## 4) Функціональний мінімум
- Catalog:
  - GET /api/catalog
  - GET /api/catalog/{id}
  - POST /api/catalog
- Customer:
  - GET /api/customer
  - GET /api/customer/{id}
  - POST /api/customer
- Order:
  - GET /api/order
  - GET /api/order/{id}
  - POST /api/order

## 5) Бізнес-логіка предметної області (ніша одягу і кросівок)
- Замовлення складається з позицій товарів (outerwear/sneakers).
- Для товару важливі розмір, колір, сезонність.
- Під час створення замовлення перевіряється наявність товару на складі.
- Сума замовлення розраховується як сума позицій.

## 6) Definition of Done (для зони Учасника 2)
- Catalog/Customer/Order endpoint-и працюють та узгоджені зі спільним стилем API.
- Є валідація вхідних DTO та коректна обробка помилок.
- Код у зоні відповідальності Учасника 2 повністю покритий тестами в різних сценаріях: позитивні, негативні, edge-case, інтеграційні бізнес-потоки.
- Інтеграція з Product модулем від Учасника 1 не ламає контракт.
