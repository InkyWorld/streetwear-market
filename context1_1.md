# Context 1.1 — Учасник 1 (Backend/API Lead)

## 1) Загальний контекст практичної
- Практична робота №1: реалізація e-commerce Web API з класичними архітектурними підходами.
- Початкові вимоги сформульовані під .NET 10, але для цього проєкту обрана реалізація на Python.
- Обрана ніша: ринок верхнього одягу та кросівок.
- Формат роботи: команда з 2 учасників.
- Ціль: отримати робочий REST API прототип онлайн-магазину з повним CRUD для Product і базовими операціями для інших сутностей.

## 2) Узгоджена технічна рамка (Python-адаптація)
- Мова: Python 3.12+.
- Віртуальне оточення та менеджер пакетів: uv (обов'язково для всієї команди).
- Фреймворк: FastAPI.
- ORM: SQLAlchemy 2.0 (async).
- Міграції: Alembic.
- БД: PostgreSQL (локально через Docker або локальна інсталяція).
- Валідація/DTO: Pydantic v2 (request/response schemas).
- Тестування: pytest + httpx + pytest-asyncio.
- Інструменти якості: ruff + black + mypy (базовий режим).

## 3) Архітектурна модель
- Шарова архітектура:
  - API layer (routers/controllers)
  - Service layer (бізнес-логіка)
  - Repository layer (доступ до даних)
  - Domain/Data layer (моделі, зв'язки)
- Принципи:
  - SOLID (особливо SRP, DIP, OCP у сервісах та репозиторіях)
  - Dependency Injection через FastAPI Depends
  - DTO-підхід: окремі схеми для create/update/read

## 4) Домен проєкту
Мінімальні сутності:
- Product
- Catalog (або Category як каталог)
- Customer
- Order

Додатково для ніші:
- Brand
- SizeVariant (розмір/наявність)

Ключові поля Product:
- id, name, sku, description
- category_id, brand_id
- price, currency
- size, color
- season (SS/AW)
- in_stock, created_at, updated_at

## 5) Мінімальні API-вимоги (що обов'язково має бути реалізовано)
- Product (повний CRUD):
  - GET /api/product/{id}
  - POST /api/product
  - PUT /api/product/{id}
  - DELETE /api/product/{id}
- Catalog/Category:
  - GET /api/catalog
  - GET /api/catalog/{id}
  - POST /api/catalog
- Для інших сутностей: мінімум GET (list/by id) + POST.

## 6) Роль Учасника 1
Учасник 1 відповідає за:
- базовий каркас проєкту;
- архітектурний скелет (layers, interfaces);
- Product CRUD end-to-end;
- інтеграцію з БД та міграції;
- стандарти API (коди відповідей, помилки, DTO);
- основу для тестів API.

Послідовність виконання:
- Спочатку повний блок робіт виконує Учасник 1.
- Лише після завершення блоку Учасника 1 до наступної половини робіт переходить Учасник 2.

## 7) Definition of Done (для зони Учасника 1)
- Product CRUD працює стабільно і протестований.
- Код у зоні відповідальності Учасника 1 повністю покритий тестами в різних сценаріях: позитивні, негативні, edge-case, помилки валідації та конфлікти даних.
- Є міграції, що створюють усі обов'язкові таблиці.
- Розділення на шари дотримано, немає прямого доступу API-layer до ORM поза сервісами/репозиторіями.
- OpenAPI/Swagger генерується та відображає основні endpoint.
- Код проходить lint/format/type-check на базовому рівні.
